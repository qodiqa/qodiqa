Abstract
Abstract

Artificial intelligence systems routinely operationalize data at scale while Consent remains external to runtime execution, captured in static terms, batch permissions, or policy documents that do not deterministically constrain model invocation at the moment of use. The resulting infrastructure gap is structural: Authorization governs system access, but no machine-verifiable mechanism evaluates, at the execution boundary, whether the specific action is legitimate under present Consent constraints.

This document specifies QODIQA Core — the minimal adoptable runtime Consent enforcement layer, derived from the QODIQA 68-Point Enforcement Framework. QODIQA Core defines twelve Control Points, a deterministic enforcement lifecycle, Fail-Closed semantics, and tamper-evident Decision Artifacts sufficient to make Consent enforceable as a runtime control plane without requiring model retraining or infrastructure replacement.

The standard is intended for adoption by enterprises, public institutions, and AI infrastructure providers within a 30–60 day integration horizon. It is the entry-level Conformance tier of a unified QODIQA institutional corpus and constitutes the foundational layer from which full 68-point Conformance is built.

Section 1

#Executive Summary

1.1 Problem Statement: Unenforced Consent in AI Execution Paths

AI systems operationalize data at scale while Consent remains external to runtime execution, captured via static terms, batch permissions, or informal policy assertions that do not deterministically constrain model invocation at the moment of use. This produces an infrastructure gap: the technical act of calling an AI system proceeds under Authorization: API keys, role-based access controls, vendor entitlements, but without a machine-verifiable, context-bound evaluation of whether the specific action is legitimate under present Consent constraints encompassing purpose, scope, time, jurisdiction, and Revocation state.

QODIQA's foundational position is that Consent is a first-class infrastructure primitive: machine-readable, replayable, and enforced as a Gating Function before execution. The practical deficiency is not the absence of policy documents; it is the absence of Deterministic Enforcement at the decision boundary where an AI system is invoked.

1.2 The Systemic Infrastructure Gap

The absence of runtime Consent enforcement is not an edge-case deficiency. It is a structural failure class with predictable scaling properties.

Contemporary AI governance frameworks establish obligations: documentation requirements, data processing agreements, access control policies, audit procedures. These instruments are necessary and insufficient at the execution boundary. A policy document that describes permitted purposes does not prevent an AI system from acting on an impermissible purpose. An audit log that records what happened does not prevent its occurrence. Authorization that grants system access does not evaluate whether the specific action under the specific context is legitimate at the moment it is taken.

This is a missing runtime invariant, not a missing policy but a missing enforcement layer. Current AI governance architecture provides controls that observe or describe behavior. It lacks controls that deterministically gate it.

The absence of Consent enforcement at runtime is structurally analogous to operating a cryptographic system without key verification: authentication exists at configuration time, but verification is absent at the moment it matters. In both cases, the gap is not philosophical; it is architectural.

This gap scales adversely. In single-model, single-purpose deployments, the risk is bounded. In multi-agent architectures where models call other models, where orchestration layers route requests across vendors, and where a single authorization event is reused across hundreds of downstream actions, the gap becomes a systemic risk amplifier.

The only architecturally sound response is to treat runtime Consent enforcement as infrastructure: present at every boundary, deterministic in behavior, and independent of the intelligence of the system it governs.

1.3 Runtime Enforcement as the Non-Substitutable Control Layer

Training-phase controls, including dataset audits, provenance records, pre-training permissions, are necessary and insufficient for operational governance because: (a) Consent changes over time through Revocation or expiry; (b) purpose drift occurs at inference time independent of training-phase Authorization basis; and (c) deployment patterns route prompts and content across systems and vendors where retrospective governance cannot reconstruct what was permitted at the moment of use.

Runtime is the only boundary where purpose, actor identity, jurisdiction, and Revocation state are simultaneously present, evaluable, and current. Controls applied at any other lifecycle point cannot substitute for enforcement at execution. QODIQA Core positions its twelve Control Points at this boundary and at no other.

1.4 Why a Minimal Standard Is Required

Enterprises and public institutions require a practical, low-friction adoption layer that can be implemented within 30–60 days, does not require model retraining, and does not require rewriting existing AI infrastructure. A minimal standard is necessary to ensure: (i) interoperability across heterogeneous AI stacks; (ii) Deterministic Governance for regulated and contractual contexts; and (iii) a clear Conformance boundary that can be audited and independently certified.

1.5 Scope: What QODIQA Core Covers

  • A closed, explicit intent declaration interface prohibiting inferred intent
  • Cryptographically verifiable Consent Tokens and Deterministic validation logic
  • Purpose- and scope-bound Policy Evaluation with Fail-Closed semantics
  • Revocation and expiry enforcement at runtime with defined Conflict Resolution Rules
  • Immutable Decision Artifacts and Audit Logging enabling independent Replay Verification

1.6 Explicit Exclusions

QODIQA Core does not define or require: model retraining or weight alteration; content inspection, semantic classification, or probabilistic risk scoring as part of the enforcement decision; economic rights or royalty distribution infrastructure; full lifecycle controls for training and fine-tuning governance; or user interfaces, dispute workflows, or appeals dashboards. The enforcement layer decides predictably, not interpretively.

1.7 Immediate Adoption Value

QODIQA Core makes Consent enforceable as a Deterministic runtime control plane, producing reproducible Decision Artifacts and tamper-evident Audit Logs that support independent Replay Verification. Adoption reduces ambiguity at the execution boundary, eliminates purpose drift, and establishes institutional defensibility without changes to deployed AI models.

#Normative Scope and Conformance

2.1 Normative Language

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in BCP 14 [RFC 2119] [RFC 8174]. Non-implementation of any MUST-level requirement constitutes non-Conformance with this standard across all profiles. Deviation from any SHOULD-level requirement MUST be explicitly documented with written justification and recorded in the Conformance Statement (Section 13).

2.2 Scope of Applicability

This standard applies to AI systems, including large language model systems, inference APIs, and tool-using agent runtimes, where an organization requires runtime enforcement of Consent constraints over AI actions that use, transform, generate, disclose, or otherwise operationalize data subject to Consent conditions. A Conformant implementation requires that a deterministic Enforcement Gateway be positioned on the request path before model invocation. This standard is model-agnostic and requires no access to model internals, weight parameters, or training artifacts.

2.3 System Boundary

A Conformant QODIQA Core implementation governs the decision boundary between a Calling System (application, service, or agent) requesting an AI action, and an AI System (model endpoint, orchestration service, or tool-using agent runtime) that would execute that action. The Enforcement Gateway MUST operate as a pre-execution gate and MUST be capable of failing closed on any missing, malformed, or unresolvable input.

2.4 Conformant Implementation

A Conformant implementation MUST: require explicit intent declarations via a closed DecisionRequest contract; validate Consent state at runtime including expiry and Revocation prior to execution; enforce purpose- and scope-bound Policy deterministically with Fail-Closed behavior on missing, malformed, or unavailable inputs; and emit Decision Artifacts and Audit Records sufficient for independent Replay Verification.

A non-Conformant system is any system that: infers or approximates intent in the absence of explicit declaration; treats Consent enforcement as an advisory check that can be bypassed; defaults to allow on incomplete, inconsistent, or unresolvable context; or cannot produce a tamper-evident Audit Trail traceable to a Frozen Evaluation Context.

Section 3

#Threat and Risk Model

3.1 Authorization vs. Consent: Formal Distinction

Defined Terms: Adopted from QODIQA Whitepaper Section 3.3
Authorization Standing technical permission granted to an actor or system to access a resource or perform a class of operations. Expressed through roles, access control lists, API keys, or service identities. Generally static, coarse-grained, and context-independent. Answers: Is this actor permitted to perform this class of operation?
Consent Contextual, conditional legitimacy for a specific action under explicitly declared purpose, scope, time constraints, jurisdictional binding, and current Revocation state. Granular, time-bounded, and revocable. Answers: Is this specific action legitimate under present conditions?

A system may hold Authorization for a class of operations and still lack valid Consent for a specific action within that class. Runtime enforcement MUST evaluate Consent independently of Authorization. The presence of Authorization does not satisfy the Consent requirement.

3.3 Runtime vs. Training-Phase: Irreducible Distinction

Training-phase governance addresses: What data was used to construct model capabilities, and under what conditions? Runtime governance addresses: What actions are performed now, by which actor, for which purpose, under which present Consent state? These are non-substitutable control layers. Training-phase controls cannot substitute runtime enforcement because: (a) Revocation and expiry affect Consent validity independently of training provenance; (b) purpose drift emerges at inference time without correlation to the training-phase Authorization basis; and (c) multi-vendor and agentic deployment chains introduce runtime actors and data flows that were not present or knowable at training time.

3.4 Liability Surface and Evidentiary Posture

The institutional liability surface expands in direct proportion to the inability to demonstrate with objective, reproducible evidence: what the AI system was instructed to perform; the Policy Version under which the enforcement decision was made; the Consent state that applied at the precise moment of the decision; and whether Revocation and expiry were evaluated and enforced.

Without deterministic Decision Artifacts, an organization cannot objectively demonstrate lawful runtime behavior. Policy documents, intent statements, and log entries do not constitute proof that a specific action was authorized under a specific Consent state at a specific time. QODIQA's "proof over explanation" posture defines institutional defensibility as the ability to reproduce decisions from a Frozen Evaluation Context.

4.3 Allow and Deny Decision Semantics

ALLOW is issued if and only if all conditions are simultaneously satisfied: the intent declaration is explicit and structurally complete; Consent Token cryptographic integrity is verified; the Consent Token is ACTIVE for the requested purpose, scope, jurisdictional binding, and data categories; all applicable Policy conditions are satisfied; and the Decision Artifact and Audit Log entry are successfully generated and written.

DENY MUST be issued on any of: missing or malformed intent fields; invalid or unverifiable Consent Token; expired, revoked, or unresolvable Consent state; purpose mismatch; scope violation; jurisdictional constraint violation; Policy evaluation failure; Decision Artifact generation failure; or Audit Log write failure. No partial execution occurs on any DENY path.

Section 6

#Formal Determinism and Replay Invariants

6.1 Frozen Evaluation Context: Formal Definition

The Frozen Evaluation Context is the minimal immutable input set from which any Conformant enforcement decision can be independently reproduced.

Definition: Frozen Evaluation Context (FEC)
FEC = (R̂, S, Vp, Vr) R̂ = canonicalize(R): canonical serialization of the DecisionRequest R, deterministically produced from its required fields using a specified canonicalization algorithm (e.g., canonical JSON, RFC 8785).

S = consent_state_ref: an immutable reference to the resolved Consent state snapshot at evaluation time. S is not the live Consent store; it is the state as resolved and frozen at the moment of evaluation.

Vp = policy_version: the version identifier of the Policy document in effect at evaluation time. References an immutable, versioned policy artifact.

Vr = ruleset_version: the version identifier of the Ruleset applied during Policy Evaluation.

6.2 Formal Evaluation Function

Policy Evaluation is defined as a total, deterministic function over the Frozen Evaluation Context:

Definition: Evaluation Function
f : FEC → {ALLOW, DENY} decision = f(R̂, S, Vp, Vr)

where f is Total (defined for all well-formed FEC inputs), Deterministic (f(C) = f(C) for all C), Pure (produces no observable side effects), and Time-independent (does not take the current time as an input beyond the frozen context).

6.3 Determinism Invariant

Replay Invariant: For all frozen evaluation contexts C and for all Conformant implementations I₁, I₂ operating over the same policy and ruleset versions: I₁.evaluate(C) = I₂.evaluate(C) = I₁.replay(C). A decision that cannot be replayed to the same outcome cannot be proven. A Conformant implementation that fails this invariant is incorrect.

6.4 Non-Determinism Exclusion Clause

The following inputs and behaviors are explicitly prohibited during Policy Evaluation:

Prohibited Input / BehaviorRequirement
RandomnessNo calls to random number generators, probabilistic functions, or stochastic sampling
External service callsNo network calls, database reads, cache lookups, or API invocations during evaluation
Current time accessNo access to wall-clock time or system time during evaluation. Temporal bounds were resolved in QCC-05 and are encoded in S
Mutable configurationNo reads from mutable configuration stores, feature flags, or dynamic policy sources
Environment-dependent behaviorEvaluation outcome MUST NOT depend on hardware, OS, locale, timezone, or execution environment state
Side effectsf MUST NOT write to any external store, emit events, or mutate any shared state during execution

7.3 Normative Enforcement Pseudocode

Enforcement Decision Function — Normative ReferencePSEUDOCODE
function decide(request, token):
  now ← gateway_synchronized_time()  // NTP-verified source

  // QCC-01 · Closed intent contract
  if ¬validate_intent_schema(request):
    return deny("INTENT_INVALID", null)

  // QCC-02 · Cryptographic token verification
  if token = null:    return deny("CONSENT_TOKEN_MISSING", null)
  if ¬verify_signature(token, trust_anchor):
    return deny("CONSENT_SIGNATURE_INVALID", null)

  // QCC-05 · Temporal validity
  if now > token.expires_at:  return deny("CONSENT_EXPIRED", null)
  if token.nbf ∧ now < token.nbf:  return deny("CONSENT_NOT_YET_VALID", null)

  // QCC-06 · Revocation — fail-closed on registry unavailability (R-01)
  rev ← revocation_registry.lookup(token.token_id)
  if rev = REVOKED:         return deny("CONSENT_REVOKED", null)
  if rev = UNKNOWN ∨ registry_unavailable():
    return deny("REVOCATION_UNRESOLVED", null)

  // QCC-03 · Purpose binding — exact match
  if request.purpose ∉ token.purposes:
    return deny("PURPOSE_NOT_AUTHORIZED", null)

  // QCC-04 · Scope binding — strict set containment
  if request.action ∉ token.scope.actions:
    return deny("ACTION_OUT_OF_SCOPE", null)

  // QCC-07 + QCC-08 · Deterministic policy evaluation, fail-closed
  frozen ← freeze(request, token.token_id, policy_version(), ruleset_version())
  try:
    decision ← policy_engine.evaluate(frozen)
  except:
    return deny("POLICY_ENGINE_FAULT", null)

  // ALLOW path — artifact and audit write MUST precede execution
  artifact ← generate_artifact(frozen, ALLOW)  // QCC-09
  if artifact = null: return deny("ARTIFACT_GENERATION_FAILED", null)

  if audit_write(artifact, ALLOW) = FAILURE:   // QCC-10 — write-before-execute
    return deny("AUDIT_WRITE_FAILED", null)

  return allow(artifact)

Section 8

#Interoperability and Ecosystem Compatibility

8.1 Interoperability Posture

QODIQA Core is implementable by independent vendors, open-source projects, and institutional operators without weakening enforcement semantics. Interoperability does not require shared infrastructure; it requires semantic equivalence of enforcement behavior under the same Frozen Evaluation Context.

Two independent implementations are interoperable at the enforcement layer if and only if: for any valid DecisionRequest and Consent Token pair, both reach the same decision outcome; both produce artifact IDs recomputable from the same frozen context; and both reject the same classes of invalid or incomplete inputs under Fail-Closed semantics.

8.2 Vocabulary Registry Governance

Interoperability at the intent layer depends on shared enumerated vocabularies for purpose codes, action codes, and data type codes. The QODIQA governance authority MUST maintain versioned, published registries for these vocabularies. Implementations MUST declare the vocabulary version in use in their Conformance Statement.

RegistryDescription
Purpose Code RegistryVersioned enumeration of permitted purpose identifiers
Action Code RegistryVersioned enumeration of permitted action identifiers; structured as a hierarchical namespace (e.g., inference.generate_text)
Data Type Code RegistryVersioned enumeration of data category identifiers
Reason Code RegistryVersioned enumeration of denial reason codes; implementations MUST map internal error states to registered reason codes

8.3 Third-Party Implementation Model

Any party may implement a Conformant QODIQA Core enforcement gateway without license from QODIQA, provided the implementation satisfies all MUST-level requirements and declares a Conformance Profile in accordance with Section 13. Third-party implementations are subject to the same Conformance test suite as any other implementation.

8.4 Compatibility with Existing AI Stacks

QODIQA Core is compatible with vendor LLM APIs, open-source inference servers, agent frameworks, retrieval-augmented generation stacks, and tool-calling runtimes, provided a deterministic Enforcement Gateway can be positioned before model invocation on the critical path. QODIQA Core does not require access to model weights, internals, or training artifacts.

10.1 GDPR Consent Principles

The General Data Protection Regulation requires Consent to be freely given, specific, informed, and unambiguous; demonstrability of Consent is required; and withdrawal must be as easy as giving Consent. QODIQA Core provides technical mechanisms that support demonstrability of these requirements:

10.2 EU AI Act

The EU AI Act imposes record-keeping and logging requirements for high-risk AI systems and deployer obligations related to log retention, transparency, and instructions for use. QODIQA Core provides technical mechanisms supporting demonstrability of alignment: deterministic logs and tamper-evident Audit Trails (QCC-10) provide structured evidence supporting traceability and monitoring obligations; structured Decision Artifacts (QCC-09) provide context supporting deployer documentation requirements.

Institutional Closing Statement

#Institutional Closing Statement

QODIQA Core defines a minimal Consent enforcement layer that is intentionally narrow in scope. It does not interpret law, inspect content semantics, or modify AI model behavior through probabilistic governance. Its function is infrastructural: placing Deterministic Consent enforcement on the runtime execution path as a precondition to AI invocation, and producing reproducible Decision Artifacts that support independent Replay Verification.

This minimalism is not a reduction of the underlying thesis. It is the operational boundary required to make Consent enforceable in complex, heterogeneous AI systems without relying on retrospective narrative, probabilistic judgment, or assumption about prior state. The Frozen Evaluation Context requirement, the Replay Invariant, the Fail-Closed semantics, and the tamper-evident audit architecture constitute the minimum viable infrastructure layer through which Consent graduates from declaration to proof.

Where Consent must be demonstrable, deterministic, and auditable:

not a statement of intent, but a verifiable runtime fact; a Consent Enforcement Gateway is a necessary technical instrument at every AI execution boundary.

QODIQA Core defines that instrument.

The QODIQA Whitepaper establishes the conceptual and philosophical foundation. The 68-Point Enforcement Framework establishes the full governance architecture. QODIQA Core is the adoptable, testable, and certifiable entry point into that architecture, the first layer any organization can deploy, and the foundation upon which full institutional Conformance is built.

The runtime Consent enforcement gap is not a problem that resolves through policy refinement, documentation improvements, or organizational discipline. It is a structural defect in AI infrastructure. QODIQA Core is the structural remedy: deterministic, verifiable, and enforceable by design.
