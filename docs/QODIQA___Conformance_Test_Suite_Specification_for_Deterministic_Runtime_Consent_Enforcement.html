Abstract

This document is the formally complete revision of the QODIQA Conformance Test Suite Specification. Version 1.0 elevates the specification from laboratory-grade test protocol to hyperscaler-grade deterministic verification standard. All structural gaps present in prior drafts have been closed: QCC-05 through QCC-12 now carry full test card granularity; cross-architecture and runtime determinism validation is formally specified; a deterministic gateway state machine model is defined; a complete control-to-test coverage mapping matrix is provided; performance benchmark methodology is specified to deterministic standards; and canonical serialization requirements are normatively defined.

All constraints are preserved without modification. No existing control semantics have been altered. No probabilistic tolerance has been introduced. No fail-closed requirements have been relaxed. This version adds formal completeness; it does not substitute it for rigor.

Conformance assessed under this specification requires satisfaction of all mandatory test cases across all twelve control points, with reproducible evidence artifacts, architecture-pair determinism validation, and a sealed evidence bundle verifiable by an independent auditing body operating under ISO/IEC 17025 principles.

Section 1

#Purpose and Scope

1.1 Purpose

This Conformance Test Suite Specification (CTSS) defines the authoritative testing methodology by which a System Under Test (SUT) is assessed for compliance with the QODIQA normative control corpus. The objective of this specification is to transform QODIQA from a normatively defined standard into a deterministically testable and independently verifiable system. Each test defined herein corresponds to one or more normative requirements drawn from the QODIQA Core Standard and the 68-Point Enforcement Framework. Tests yield binary pass/fail determinations from reproducible, documented procedures. No subjective evaluation is permitted.

1.2 Relationship to Normative Standards

Source StandardIdentifierRole in This Specification
QODIQA — Core Standard for Deterministic Runtime Consent Enforcement Version 1.0QODIQA-CORE-v1.0Primary normative source. Defines QCC-01 through QCC-12.
QODIQA — 68-Point Enforcement Framework for Deterministic Runtime Consent Enforcement Version 1.0QODIQA-68P-v3.0Supplementary normative source. Enforcement points EP-01 through EP-68.
QODIQA — Certification Framework for Deterministic Runtime Consent Enforcement Version 1.0QODIQA-CERT-v1.0Defines certification tiers and evidence requirements.
QODIQA — Security and Cryptographic Profile for Runtime Consent Enforcement Version 1.0QODIQA-SCP-v1.1Cryptographic algorithm requirements for Sections 8, 9, 12, and 15.

1.3 System Under Test Boundary

The SUT boundary is strictly the QODIQA Gateway Enforcement Layer: the runtime component responsible for evaluating consent tokens, enforcing policy decisions, generating execution artifacts, and writing to the audit ledger.

  • Consent token validation and signature verification logic
  • Purpose and intent field parsing and enforcement
  • Policy engine evaluation and DENY/PERMIT decision emission
  • Revocation registry query and response handling
  • Execution artifact generation and hash computation
  • Audit ledger write operations and hash chain maintenance
  • Fail-closed behavior under all error and ambiguity conditions

1.4 Explicit Exclusions

  • Model internals, weights, or probabilistic inference behaviors
  • Training data pipelines or dataset provenance systems
  • Application-layer orchestration above the gateway
  • Network infrastructure external to the SUT boundary
  • End-user consent collection interfaces
  • Governance policy authoring workflows
Section 2

#Conformance Testing Principles

Principle DefinitionsTesting Axioms · QTS-P-01 through QTS-P-06
QTS-P-01 · Determinism Invariant For any frozen, canonically serialized context C, the SUT MUST produce identical output across all executions. A single divergent output across N replay iterations constitutes non-conformance.
QTS-P-02 · Fail-Closed Invariant In all conditions of ambiguity, error, incomplete context, or resource unavailability, the SUT MUST emit a DENY decision. Any PERMIT decision under ambiguous or degraded conditions constitutes critical non-conformance.
QTS-P-03 · Reproducibility Requirement Every test MUST be independently reproducible by an external auditor from the documented procedure and captured evidence artifacts alone. Tests that cannot be reproduced from artifacts are invalid.
QTS-P-04 · Negative Test Priority Negative test cases — those designed to elicit DENY, rejection, or error behavior — are considered primary conformance indicators. Positive path validation is secondary to negative boundary validation.
QTS-P-05 · No Probabilistic Allowance No statistical pass rate is acceptable. Conformance is binary. A SUT that passes 99 of 100 iterations of a determinism test is non-conformant.
QTS-P-06 · Evidence Capture Requirement Every test execution MUST capture structured evidence artifacts as defined in Section 16. Test results unsupported by captured evidence SHALL be treated as inconclusive.

2.1 Formal Determinism Invariant

Formal InvariantQTS-P-01 · Determinism Constraint
-- Determinism Invariant: QTS-P-01
INVARIANT Determinism:
  FORALL C : FrozenConsentContext,
  FORALL i, j : ExecutionRun,
    evaluate(C, i) == evaluate(C, j)

-- Failure condition:
--   EXISTS i, j such that evaluate(C, i) ≠ evaluate(C, j)
--   => SUT is NON-CONFORMANT under QTS-P-01

Section 5

#Determinism Verification Protocol

5.1 Frozen Context Capture Procedure

NormativeFrozen Context Structure · QTS-FC-01
consent_token_hashSHA-256 of raw serialized consent token. Computed before any field parsing.
request_context_hashSHA-256 of canonically serialized request context per QCC-06 and Section 15.
policy_version_idExact policy version string active at time of evaluation. MUST include patch version component.
registry_state_snapshotDeterministic fixture identifier for the registry state seeded for this test run.
evaluation_timestampUnix epoch timestamp at moment of evaluation initiation, millisecond precision.
frozen_context_idSHA-256 of concatenation of all above fields in canonical order. Used as replay identifier.

5.2 Replay Testing Procedure

  • Capture frozen context record and assign frozen_context_id
  • Execute evaluation against frozen context. Record decision D₁ and full output log.
  • Reset SUT to clean state. Verify registry fixture re-seeded to identical deterministic state.
  • Repeat for minimum N = 10 independent iterations yielding D₁ through D₁₀.
  • If EXISTS i, j: Dᵢ ≠ Dⱼ, the SUT is non-conformant under QTS-P-01.
  • Package all iteration logs as a replay evidence bundle per Section 16.

Formal InvariantQTS-CA-01 · Cross-Architecture Determinism
-- Cross-Architecture Determinism Invariant: QTS-CA-01
-- For identical FrozenContext C across any two architecture targets A and B:

INVARIANT CrossArchitectureDeterminism:
  FORALL C  : FrozenConsentContext,
  FORALL A, B : Architecture -- e.g. x86_64, ARM64, ARMv7, RISC-V

-- Where evaluate_X(C) returns:

Section 13

#Formal Gateway State Machine Model

The QODIQA Gateway Enforcement Layer operates as a deterministic finite state machine. Every request transitions through a fixed sequence of states. Any failure or ambiguity at any state MUST transition to the DENY_EMIT terminal state. No state transition may be skipped. Each transition has defined pre-conditions, post-conditions, failure paths, determinism requirements, and mandatory log entries.

13.1 State Transition Formal Invariant

Formal ModelQTS-SM-01 · Gateway State Sequence
STATES := [
  INTAKE, STRUCTURAL_VALIDATE, SIGNATURE_VERIFY,
  TEMPORAL_VALIDATE, REVOCATION_CHECK, POLICY_EVALUATE,
  ARTIFACT_GENERATE, AUDIT_WRITE, DECISION_EMIT
]

INVARIANT StateMonotonicity:
  -- Normal path: each state reached at most once, in defined order
  FORALL r : Request, s ∈ STATES :
    reached(r, s) ⇒ ¬revisited(r, s)

INVARIANT FailClosed:
  -- Any failure at any state MUST produce DENY
  FORALL r : Request, s ∈ STATES :
    failure_at(r, s) ⇒ terminal_decision(r) == DENY

INVARIANT NoSkip:
  -- States MUST be traversed in sequence; no skipping
  FORALL r : Request :
    sequence(r) ⊆ ordered_prefix(STATES)

13.2 State Transition Definitions

INTAKE
Pre-conditionRequest received by gateway process; bytes fully buffered
ActionExtract Authorization header. Bind policy_version_id at intake timestamp. Assign request_id.
Post-conditionToken bytes available. Policy version pinned. Request ID assigned.
Failure path→ DENY_EMIT: missing Authorization header, oversized payload, parse error
Log entry{ state: INTAKE, request_id, policy_version_id, timestamp }
STRUCTURAL_VALIDATE
Pre-conditionToken bytes available from INTAKE
ActionParse token. Verify all required claims present and non-empty. Verify purpose code is registered. Reject unknown fields if strict mode.
Post-conditionAll required fields validated. Claim values available for subsequent states.
Failure path→ DENY_EMIT: malformed token, missing claim, empty claim value, unknown purpose code
Log entry{ state: STRUCTURAL_VALIDATE, result: PASS|FAIL, failure_reason? }
SIGNATURE_VERIFY
Pre-conditionStructurally valid token available. Trust anchor registry accessible.
ActionResolve key ID against trust anchor registry. Verify algorithm meets minimum policy. Verify cryptographic signature.
Post-conditionSignature verified. Key ID confirmed as non-revoked trust anchor.
Failure path→ DENY_EMIT: signature invalid, unknown key, revoked key, weak algorithm, alg:none
Log entry{ state: SIGNATURE_VERIFY, key_id, algorithm, result: PASS|FAIL }
TEMPORAL_VALIDATE
Pre-conditionSignature verified. Clock source available.
ActionVerify exp is future. Verify nbf is past. Verify iat is not future. Verify lifetime does not exceed policy maximum.
Post-conditionToken confirmed valid within temporal window.
Failure path→ DENY_EMIT: expired, not-yet-valid, missing exp, lifetime exceeded, future iat
Log entry{ state: TEMPORAL_VALIDATE, exp, nbf, iat, result: PASS|FAIL }
REVOCATION_CHECK
Pre-conditionTemporally valid token. Revocation registry reachable within timeout.
ActionQuery revocation registry for token ID. Await confirmed response within timeout. Validate response authenticity.
Post-conditionRegistry returned CONFIRMED_NOT_REVOKED within timeout.
Failure path→ DENY_EMIT: revoked, absent, timeout, malformed response, stale cache, partition
Log entry{ state: REVOCATION_CHECK, registry_status, latency_ms, result: PASS|FAIL }
POLICY_EVALUATE
Pre-conditionToken not revoked. Context hash computed and bound. Policy version pinned from INTAKE.
ActionEvaluate request against pinned policy version. Apply conflict resolution (DENY wins). Emit PERMIT or DENY with matched rule ID.
Post-conditionDeterministic PERMIT or DENY with matched rule ID and policy version reference.
Failure path→ DENY_EMIT: engine failure, null decision, no rule matched (default-deny), DENY rule matched
Log entry{ state: POLICY_EVALUATE, decision, matched_rule_id, policy_version_id }
ARTIFACT_GENERATE
Pre-conditionPERMIT decision from POLICY_EVALUATE. Hash engine available.
ActionConstruct artifact with minimised fields. Compute and embed artifact_hash. Bind context_hash and policy_version_id.
Post-conditionArtifact produced with verified hash. No raw sensitive data present.
Failure path→ DENY_EMIT: generator failure, hash failure, missing required field in artifact
Log entry{ state: ARTIFACT_GENERATE, artifact_hash, context_hash, result: PASS|FAIL }
AUDIT_WRITE
Pre-conditionDecision (PERMIT or DENY) available. Artifact available (PERMIT only). Audit ledger writable.
ActionAppend decision record to hash-chained ledger. Await write confirmation. Verify chain integrity for new entry.
Post-conditionDecision record durably written. Hash chain intact.
Failure path→ DENY_EMIT (if PERMIT pending): write failure, timeout, chain verification failure
Log entry{ state: AUDIT_WRITE, ledger_entry_id, chain_hash, result: PASS|FAIL }
DECISION_EMIT
Pre-conditionAUDIT_WRITE confirmed. Decision finalized.
ActionEmit PERMIT or DENY response to caller with decision metadata.
Post-conditionResponse delivered to caller. Terminal state reached.
Failure pathNone — terminal state. DENY emitted on any unresolved failure from prior states.
Log entry{ state: DECISION_EMIT, final_decision, request_id, total_latency_ms }

Section 15

#Canonical Serialization Requirements

Determinism in hash computation and cross-architecture equivalence depends on canonical serialization. Any serialization deviation that produces a different byte sequence for logically identical data constitutes a determinism failure. The following requirements are normative and non-negotiable.

NormativeCanonical Serialization Specification · QTS-CS-01
Character EncodingUTF-8 encoding ONLY. No UTF-16, UTF-32, or BOM-prefixed encodings are permitted at any serialization boundary.
Object Key OrderingAll JSON object keys MUST be sorted in lexicographic byte order (ascending). Implementations MUST NOT rely on insertion order or language-native dictionary ordering.
WhitespaceNo insignificant whitespace. No newlines, spaces, or tabs outside of string values. Canonical form is compact JSON with no padding.
Integer RepresentationIntegers serialized as decimal without leading zeros. No scientific notation. No floating-point representation for integer-valued quantities. Range: signed 64-bit integer.
Floating-PointFloating-point values MUST NOT appear in any field that participates in hash computation. If unavoidable, serialize using IEEE 754 double-precision canonical decimal representation with no trailing zeros.
Unicode NormalizationAll string values MUST be NFC-normalized (Unicode Normalization Form C) before serialization. Strings that are logically equivalent but NFC-distinct MUST produce identical serialized form after normalization.
Null and BooleanSerialize as lowercase literals: null, true, false. No alternatives permitted.
Array OrderingArray element order is semantically significant and MUST be preserved exactly. Arrays MUST NOT be re-sorted during serialization.
Byte SequencesRaw byte sequences MUST be encoded as lowercase hexadecimal strings. No Base64 or Base64url encoding in hash-participating fields.

Section 18

#Institutional Closing Statement

This specification defines the complete and authoritative conformance testing methodology for QODIQA-compliant enforcement systems. Version 1.0 achieves formal completeness across all twelve control points, cross-architecture determinism validation, a deterministic gateway state machine model, full control-to-test coverage mapping, and deterministic benchmark methodology. The requirements herein are exhaustive with respect to the defined scope and non-negotiable in their enforcement.

Foundational Conformance Axioms

Conformance is measured through deterministic invariants. There is no probabilistic tolerance, no partial credit, and no acceptable ambiguity in enforcement outcome. A system that produces the correct decision in 99.9% of cases is not conformant.

Enforcement Invariant

Fail-closed behavior is non-negotiable. Any system that permits execution under ambiguous, degraded, or incomplete conditions has failed the most fundamental requirement of consent-bound enforcement. This cannot be remediated by subsequent logging, monitoring, or corrective action.

Architecture Invariant

Cross-architecture determinism is a first-class conformance requirement. A system that produces identical results on x86-64 but divergent results on ARM64 is non-conformant regardless of deployment context. The enforcement invariant applies uniformly across all architectures and runtime environments.

Verification Requirement

Replay reproducibility is mandatory. An enforcement decision that cannot be independently reproduced from its evidence artifacts is not a verified decision. Certification requires not only correct behavior but demonstrably correct, reproducible, architecture-independent behavior.
